# 数据验证与审核技术学习总结

## 一、统计分析方法

### 1.1 假设检验 (Hypothesis Testing)

#### 核心概念
假设检验是统计学中用于判断样本数据是否支持某个统计假设的方法。

**基本步骤：**
1. **建立假设**
   - 原假设 H₀：默认成立的假设（如：数据均值 = 基准值）
   - 备择假设 H₁：想要验证的假设（如：数据均值 ≠ 基准值）

2. **选择检验统计量**
   - Z检验：大样本(n>30)或已知总体标准差
   - T检验：小样本(n≤30)且未知总体标准差
   - 卡方检验：分类变量独立性检验

3. **确定显著性水平 α**
   - 常用值：0.05（95%置信度）、0.01（99%置信度）

4. **计算p值并决策**
   - p < α：拒绝H₀，认为差异显著
   - p ≥ α：不拒绝H₀，无足够证据表明差异显著

#### LNG数据审核应用场景
| 场景 | 检验类型 | 应用示例 |
|------|----------|----------|
| 价格合理性 | 单样本T检验 | 检验某港口LNG价格是否偏离历史均值 |
| 供应商对比 | 双样本T检验 | 比较两家供应商价格是否存在显著差异 |
| 价格相关性 | 相关性检验 | 验证LNG价格与原油价格的相关性 |

---

### 1.2 置信区间 (Confidence Interval)

#### 核心概念
置信区间表示估计值的不确定性范围，反映估计的精确度。

**计算公式：**

**均值置信区间（大样本）：**
```
CI = x̄ ± Z_(α/2) × (σ/√n)
```
其中：x̄为样本均值，σ为标准差，n为样本量，Z_(α/2)为标准正态分布分位数

**均值置信区间（小样本）：**
```
CI = x̄ ± t_(α/2, n-1) × (s/√n)
```
其中：s为样本标准差，t为t分布临界值

**常用Z值：**
- 90%置信度：Z = 1.645
- 95%置信度：Z = 1.96
- 99%置信度：Z = 2.576

#### LNG数据审核应用场景
1. **价格区间估计**：计算某区域LNG价格的95%置信区间，识别超出区间的异常报价
2. **交易量预测**：基于历史数据建立交易量的置信区间，评估实际交易合理性
3. **供应商评分**：计算供应商评分指标的置信区间，区分统计显著差异

---

### 1.3 相关性分析 (Correlation Analysis)

#### 核心概念
相关性分析用于衡量两个变量之间的线性关系强度和方向。

**皮尔逊相关系数 (Pearson r)：**
```
r = Σ[(xi - x̄)(yi - ȳ)] / √[Σ(xi - x̄)² × Σ(yi - ȳ)²]
```
- r ∈ [-1, 1]
- |r| > 0.7：强相关
- 0.3 < |r| ≤ 0.7：中等相关
- |r| ≤ 0.3：弱相关

**斯皮尔曼等级相关 (Spearman ρ)：**
- 适用于非线性关系或有序数据
- 对异常值不敏感

#### LNG数据审核应用场景
| 变量对 | 预期相关性 | 审核应用 |
|--------|-----------|----------|
| LNG价格 ↔ 原油价格 | 强正相关(r>0.8) | 检验价格联动合理性 |
| LNG价格 ↔ 天然气期货 | 强正相关 | 验证期货贴水/升水逻辑 |
| 交易量 ↔ 价格波动 | 中等相关 | 识别异常交易模式 |
| 到港量 ↔ 库存变化 | 强负相关 | 验证库存数据一致性 |

## 二、异常检测算法

### 2.1 3σ原则 (Three Sigma Rule)

#### 核心概念
基于正态分布特性，数据落在均值±3个标准差范围内的概率为99.7%。

**判定规则：**
- 正常值：μ - 3σ ≤ x ≤ μ + 3σ
- 异常值：x < μ - 3σ 或 x > μ + 3σ

**适用条件：**
- 数据近似服从正态分布
- 样本量足够大（n > 30）

**优点：**
- 计算简单，实时性好
- 可解释性强
- 阈值直观

**缺点：**
- 对非正态分布数据效果差
- 对极端值敏感（标准差会被拉大）

#### LNG数据审核应用
```python
# 伪代码示例
def detect_outliers_3sigma(data):
    mean = np.mean(data)
    std = np.std(data)
    lower_bound = mean - 3 * std
    upper_bound = mean + 3 * std
    
    outliers = [x for x in data if x < lower_bound or x > upper_bound]
    return outliers, (lower_bound, upper_bound)
```

**适用指标：**
- 日交易量
- 单船装载量
- 港口库存量
- 运输周期时长

---

### 2.2 箱线图法 (IQR Method)

#### 核心概念
基于四分位数范围(IQR)识别异常值，对非正态分布更稳健。

**计算步骤：**
1. 计算Q1（第25百分位数）和Q3（第75百分位数）
2. 计算IQR = Q3 - Q1
3. 确定边界：
   - 下界 = Q1 - 1.5 × IQR
   - 上界 = Q3 + 1.5 × IQR
4. 异常值判定：
   - 温和异常值：1.5×IQR < 距离 < 3×IQR
   - 极端异常值：距离 > 3×IQR

**优点：**
- 对异常值不敏感（基于中位数）
- 适用于偏态分布
- 可识别异常方向（过高/过低）

**缺点：**
- 小样本时四分位数不稳定
- 1.5倍系数为经验值

#### LNG数据审核应用
```python
def detect_outliers_iqr(data):
    q1 = np.percentile(data, 25)
    q3 = np.percentile(data, 75)
    iqr = q3 - q1
    
    lower_fence = q1 - 1.5 * iqr
    upper_fence = q3 + 1.5 * iqr
    
    mild_outliers = [x for x in data if 
                     (x < lower_fence and x >= q1 - 3*iqr) or 
                     (x > upper_fence and x <= q3 + 3*iqr)]
    extreme_outliers = [x for x in data if 
                        x < q1 - 3*iqr or x > q3 + 3*iqr]
    
    return mild_outliers, extreme_outliers
```

**适用指标：**
- 供应商报价分布
- 不同港口价格差异
- 运输成本分布
- 到港延迟时间

### 2.3 孤立森林算法 (Isolation Forest)

#### 核心概念
基于随机划分的无监督异常检测算法，异常点更容易被"孤立"。

**核心思想：**
- 正常数据点：需要更多次划分才能被孤立
- 异常数据点：特征值与正常数据差异大，更容易被孤立

**算法步骤：**
1. 随机选择特征和分割值构建多棵孤立树
2. 计算每个样本在所有树中的平均路径长度
3. 路径长度短的样本更可能是异常值
4. 计算异常分数：
```
s(x, n) = 2^(-E(h(x))/c(n))
```
其中：E(h(x))为平均路径长度，c(n)为标准化因子

**优点：**
- 无需标注数据（无监督）
- 处理高维数据能力强
- 对全局异常和局部异常都有效
- 时间复杂度低（O(n log n)）

**缺点：**
- 对高维稀疏数据效果下降
- 需要调参（树的数量、子采样大小）

#### LNG数据审核应用
```python
from sklearn.ensemble import IsolationForest

# 多维度异常检测
def detect_anomalies_isolation_forest(data):
    """
    data: DataFrame with features [price, volume, distance, delivery_time, ...]
    """
    clf = IsolationForest(
        n_estimators=100,      # 树的数量
        max_samples='auto',     # 子采样大小
        contamination=0.05,     # 预期异常比例
        random_state=42
    )
    
    clf.fit(data)
    predictions = clf.predict(data)  # 1:正常, -1:异常
    scores = clf.decision_function(data)  # 异常分数
    
    return predictions, scores
```

**多维度特征组合：**
| 维度 | 特征示例 |
|------|----------|
| 价格维度 | 单价、总价、单位运输成本 |
| 数量维度 | 交易量、装载量、库存量 |
| 时间维度 | 交货周期、到港延迟、报价有效期 |
| 空间维度 | 运输距离、港口位置、供应商区域 |

## 三、数据质量评估

### 3.1 完整性评估 (Completeness)

#### 核心概念
衡量数据集中缺失值的程度，反映数据的完备性。

**评估指标：**

**1. 字段完整率**
```
字段完整率 = (非空记录数 / 总记录数) × 100%
```

**2. 记录完整率**
```
记录完整率 = (完整记录数 / 总记录数) × 100%
```
（完整记录：关键字段均无缺失）

**3. 整体完整度**
```
整体完整度 = Σ(字段权重 × 字段完整率)
```

**完整性等级：**
| 完整率 | 等级 | 处理建议 |
|--------|------|----------|
| >99% | 优秀 | 正常使用 |
| 95%-99% | 良好 | 监控缺失模式 |
| 90%-95% | 一般 | 需补充数据 |
| <90% | 较差 | 暂停使用，排查原因 |

#### LNG数据审核应用

**关键字段完整性检查：**
```python
completeness_rules = {
    'transaction_id': {'required': True, 'weight': 1.0},
    'supplier_name': {'required': True, 'weight': 1.0},
    'price': {'required': True, 'weight': 1.0},
    'volume': {'required': True, 'weight': 1.0},
    'delivery_date': {'required': True, 'weight': 1.0},
    'port_name': {'required': True, 'weight': 0.8},
    'vessel_name': {'required': False, 'weight': 0.5},
    'quality_spec': {'required': False, 'weight': 0.6}
}

def assess_completeness(record, rules):
    total_weight = 0
    weighted_score = 0
    
    for field, config in rules.items():
        weight = config['weight']
        total_weight += weight
        
        if config['required'] and not record.get(field):
            return 'CRITICAL', 0  # 关键字段缺失
        elif record.get(field):
            weighted_score += weight
    
    completeness_rate = weighted_score / total_weight
    return 'PASS', completeness_rate
```

### 3.2 一致性检查 (Consistency)

#### 核心概念
验证数据内部逻辑关系是否正确，确保数据之间无矛盾。

**一致性类型：**

**1. 格式一致性**
- 日期格式统一（YYYY-MM-DD）
- 货币单位统一（USD/吨）
- 编码规范统一（供应商编码、港口编码）

**2. 逻辑一致性**
- 数值范围合理（价格>0，数量>0）
- 时间顺序合理（报价日期 ≤ 交货日期）
- 关联关系正确（供应商编码存在于主数据表）

**3. 跨系统一致性**
- 同一数据在不同系统间一致
- 汇总数据与明细数据一致

#### LNG数据审核应用

**逻辑一致性检查规则：**
```python
consistency_rules = [
    {
        'name': '价格正数检查',

        'severity': 'ERROR'

    {

        'check': lambda r: r['volume'] > 0,
        'severity': 'ERROR'
    },
    {
        'name': '日期顺序检查',
        'check': lambda r: r['quote_date'] <= r['delivery_date'],
        'severity': 'ERROR'
    },

        'name': '交货周期合理性',

```

**参考数据一致性：**
| 检查项 | 参考数据源 | 检查逻辑 |
|--------|-----------|----------|
| 供应商有效性 | 供应商主数据 | 供应商编码存在且状态为"有效" |
| 港口有效性 | 港口主数据 | 港口编码存在且支持LNG业务 |
| 价格合理性 | 历史价格区间 | 价格在历史均值±3σ范围内 |

### 3.3 时效性评估 (Timeliness)

#### 核心概念
衡量数据从产生到可用的时间延迟，反映数据的更新及时性。

**评估指标：**

**1. 数据延迟时间**
```
数据延迟 = 数据入库时间 - 数据产生时间
```

**2. 数据新鲜度**
```
新鲜度 = 1 - (当前时间 - 最后更新时间) / 最大容忍延迟
```

**3. 时效性等级：**
| 延迟 | 等级 | 适用场景 |
|------|------|----------|
| <1小时 | 实时 | 高频交易、价格监控 |
| <4小时 | 准实时 | 日报数据、库存更新 |
| <24小时 | 日更新 | 交易汇总、供应商评估 |
| >24小时 | 滞后 | 月度统计、历史分析 |

#### LNG数据审核应用

**时效性检查规则：**
```python
timeliness_rules = {
    'price_quote': {
        'max_delay_hours': 1,
        'business_hours_only': True,
        'severity': 'WARNING'
    },
    'transaction_record': {
        'max_delay_hours': 4,
        'business_hours_only': False,
        'severity': 'ERROR'
    },
    'inventory_update': {
        'max_delay_hours': 24,
        'business_hours_only': False,
        'severity': 'WARNING'
    },
    'vessel_arrival': {
        'max_delay_hours': 2,
        'business_hours_only': False,
        'severity': 'ERROR'
    }
}

def assess_timeliness(record_type, event_time, ingest_time, rules):
    rule = rules.get(record_type)
    delay_hours = (ingest_time - event_time).total_seconds() / 3600
    
    if delay_hours > rule['max_delay_hours']:
        return False, f"延迟{delay_hours:.1f}小时，超过阈值{rule['max_delay_hours']}小时"
    return True, f"延迟{delay_hours:.1f}小时，符合要求"
```

| 假设检验 | 明确H₀/H₁，理解p值含义，选择合适检验方法 |
| 置信区间 | 反映估计不确定性，用于识别异常范围 |
| 相关性分析 | 区分相关与因果，选择合适的相关系数 |
| 3σ原则 | 基于正态分布，计算简单但对分布敏感 |
| 箱线图法 | 基于分位数，对异常值稳健 |
| 孤立森林 | 无监督、多维、适合复杂异常模式 |
| 完整性 | 区分关键/非关键字段，加权评估 |
| 一致性 | 建立业务规则库，覆盖格式和逻辑 |
| 时效性 | 根据业务场景设定合理延迟阈值 |

### 5.2 实施优先级建议

**高优先级（立即实施）：**
1. 完整性检查 - 数据质量基础
2. 一致性检查 - 业务逻辑验证
3. 3σ/IQR异常检测 - 单指标异常识别

**中优先级（2-4周内）：**
1. 时效性检查 - 数据新鲜度保障
2. 相关性验证 - 价格合理性交叉验证

**低优先级（持续优化）：**
1. 孤立森林 - 复杂模式识别
2. 假设检验 - 统计显著性验证
